1. PROJECT OVERVIEW
--------------------------------------------------------------------------------

Nexa is a full-stack, domain-restricted AI assistant platform built for
Nexa Homes, a turnkey contracting and interior design company based in
Erode, India. The platform serves three distinct user roles — public
visitors (New Users), authenticated Clients, and Administrators — each
with tailored AI-powered chat experiences, dashboards, and analytics.

The system uses a locally hosted Large Language Model (LLM) combined with
a Retrieval-Augmented Generation (RAG) pipeline to answer user queries
exclusively within the company's domain, ensuring no hallucinated or
off-topic responses.

4. APPLICATION WORKFLOW
--------------------------------------------------------------------------------

4.1 PUBLIC USER FLOW
  1. Visitor lands on the Index (home) page
  2. Opens the AI chat panel — no login required
  3. Submits a question about Nexa Homes services
  4. Backend classifies intent:
     a. OUT_OF_DOMAIN → Returns polite domain-restriction message instantly
     b. COMPANY_INFO / PRICING → RAG retrieves relevant knowledge chunks →
        LLM generates context-grounded response
  5. Response streams to the user via SSE in real-time (~300ms first token)
  6. Chat interaction is logged to MongoDB (query, response, intent, sentiment)

4.2 CLIENT FLOW
  1. Client logs in via /login with email and password
  2. JWT token is issued and stored in localStorage
  3. Redirected to /dashboard (ClientDashboard)
  4. Dashboard displays:
     - Active project status, completion %, and timeline
     - Latest weekly progress update with images
     - Full update history
  5. Client uses AI chat for project-specific queries:
     - Classifier detects PROJECT_INFO intent
     - System fetches client's project data + latest update from DB
     - LLM generates personalized response grounded in real project data
  6. Multi-language support: client can select response language

4.3 ADMIN FLOW
  1. Admin logs in and is redirected to /admin (AdminDashboard)
  2. Admin can:
     - View daily query metrics and breakdown by user role
     - View aggregated sentiment analysis of all chat interactions
     - Create new projects (assigning clients and constructors)
     - Post weekly progress updates with images (up to 10 per update)
     - Manage users and constructors
  3. Constructor assignment enforces a max of 2 projects per constructor

4.4 CHAT PROCESSING PIPELINE (DETAILED)
  Request → optionalAuth middleware (extract user if token present)
         → Intent Classification (classifier.ts)
         → Prompt Assembly:
              - OUT_OF_DOMAIN  → Static rejection message
              - PROJECT_INFO   → DB lookup (project + latest update) → Client prompt
              - COMPANY_INFO   → RAG retrieval (embedding search) → New user prompt
         → LLM Generation (streaming via Ollama API)
         → SSE delivery to frontend
         → Async logging to ChatLog collection
         → Sentiment analysis on combined query + response

5. AI FEATURE DESCRIPTION
--------------------------------------------------------------------------------

5.1 RETRIEVAL-AUGMENTED GENERATION (RAG)
  The RAG pipeline ensures the AI assistant answers only from verified company
  information. It works as follows:
    - A static knowledge base of 10 semantic chunks covers topics like company
      overview, services, pricing, materials, team, vision, mission, and contact.
    - At server startup, all chunks are embedded using the nomic-embed-text model
      via Ollama's /api/embeddings endpoint (parallel warmup).
    - When a query arrives, the user's question is also embedded.
    - Cosine similarity is computed between the query vector and all chunk vectors.
    - The top-2 most similar chunks are selected and injected into the LLM prompt.
    - A keyword-based fallback exists if the embedding service is unavailable.

5.2 INTENT CLASSIFICATION
  A lightweight heuristic classifier routes queries to the appropriate handler:
    - PROJECT_INFO  : Triggered by keywords like "my project", "status", "timeline"
    - PRICING       : Triggered by "cost", "price", "budget", "how much"
    - COMPANY_INFO  : Default fallback for general queries
    - OUT_OF_DOMAIN : Triggered by blocklisted terms (politics, medical, recipes,
                      code generation, etc.) — ensures strict domain restriction
  Role-aware logic: pricing queries from clients are treated as project info.

5.3 STREAMING CHAT (SSE)
  The primary chat endpoint uses Server-Sent Events for real-time token delivery:
    - Ollama's streaming API sends tokens as newline-delimited JSON
    - Backend reads the stream and forwards each token as an SSE data event
    - Frontend renders tokens incrementally, showing the response as it generates
    - First token arrives in ~300ms, drastically reducing perceived latency
    - A final SSE event delivers metadata (intent, sentiment, execution time)

5.4 SENTIMENT ANALYSIS
  Every chat interaction undergoes sentiment classification:
    - Keywords like "great", "excellent", "happy" → POSITIVE
    - Keywords like "bad", "terrible", "slow", "angry" → NEGATIVE
    - Everything else → NEUTRAL
  Results are stored in ChatLog and aggregated for admin analytics dashboards.

5.5 MULTI-LANGUAGE SUPPORT
  The AI assistant supports 6 languages:
    - English (en), Tamil (ta), Hindi (hi), Telugu (te), Kannada (kn), Malayalam (ml)
  Language selection is done by the user in the chat panel. The selected language
  code is sent with each request, and the system prompt instructs the LLM to
  respond entirely in the chosen language, including all labels and navigation text.

5.6 DOMAIN RESTRICTION
  The system enforces strict domain boundaries to prevent misuse:
    - The intent classifier blocks out-of-domain queries at the entry point
    - System prompts explicitly instruct the LLM to only use provided context
    - If the answer is not available, the LLM directs users to contact the company
    - No general knowledge, medical, political, or code generation queries are served

5.7 PERSONALIZED CLIENT RESPONSES
  For authenticated clients, the AI provides project-specific answers:
    - Real project data (name, status, completion %, budget, location) is injected
    - The latest weekly progress update (title, description, images) is included
    - The assigned constructor's name is mentioned
    - Responses are grounded entirely in real data — no hallucination

5.8 CHAT LOGGING AND ANALYTICS
  Every interaction is logged to MongoDB with:
    - User ID (if authenticated), role, query text, full response
    - Detected intent, sentiment classification
    - Related project ID (for client queries)
    - Execution time in milliseconds
  Admin dashboard provides daily metrics and sentiment distribution charts.

6. MODEL DESCRIPTION
--------------------------------------------------------------------------------

6.1 LARGE LANGUAGE MODEL — Llama 3.2
  Provider        : Meta (served locally via Ollama)
  Model ID        : llama3.2
  Deployment      : Local (Ollama server at http://localhost:11434)
  Parameters      :
    - Temperature   : 0.1 (low randomness for factual, consistent responses)
    - Max Tokens    : 256 (num_predict — keeps responses concise)
    - Context Window: 2048 tokens (num_ctx)
  Usage           :
    - Generates all chat responses for public users and clients
    - Supports both streaming (SSE) and non-streaming modes
    - Receives structured system prompts with injected context/data
  Advantages      :
    - Runs entirely on-premise — no external API calls or data leakage
    - Zero recurring API costs
    - Full control over model behavior via prompt engineering
    - Fast inference with streaming for low-latency user experience

6.2 EMBEDDING MODEL — nomic-embed-text
  Provider        : Nomic AI (served locally via Ollama)
  Model ID        : nomic-embed-text
  Deployment      : Local (Ollama server at http://localhost:11434)
  Usage           :
    - Converts text into dense vector representations (embeddings)
    - Used for the RAG pipeline's semantic similarity search
    - Generates embeddings for both knowledge base chunks and user queries
    - Cosine similarity between vectors determines the most relevant chunks
  Performance     :
    - Embeddings are pre-warmed at server startup (parallel processing)
    - Eliminates cold-start latency on the first user query
    - Lightweight model optimized for retrieval tasks

6.3 DATA MODELS (MongoDB Collections)

  User:
    - email          : String (unique)
    - passwordHash   : String (bcrypt hashed)
    - role           : Enum — "CLIENT" | "ADMIN" | "CONSTRUCTOR"
    - name           : String
    - timestamps     : createdAt, updatedAt (auto-managed)

  Project:
    - clientId       : Reference to User
    - constructorId  : Reference to User (CONSTRUCTOR role)
    - projectName    : String
    - status         : Enum — "PLANNING" | "IN_PROGRESS" | "COMPLETED"
    - completionPercentage : Number (0–100)
    - timeline       : Array of { date, milestone, status }
    - details        : { location, budget, architect }
    - lastUpdated    : Date

  ProjectUpdate:
    - projectId      : Reference to Project (indexed)
    - title          : String
    - description    : String
    - completionPercentage : Number (0–100)
    - images         : Array of Strings (file paths)
    - postedBy       : Reference to User
    - timestamps     : createdAt, updatedAt (auto-managed)
    - Compound index : { projectId: 1, createdAt: -1 } for fast latest-update queries

  ChatLog:
    - userId         : Reference to User (optional — null for public users)
    - role           : Enum — "CLIENT" | "NEW_USER" | "ADMIN"
    - query          : String
    - response       : String
    - detectedIntent : String
    - sentiment      : Enum — "POSITIVE" | "NEUTRAL" | "NEGATIVE"
    - relatedProjectId : Reference to Project (optional)
    - executionTimeMs: Number
    - timestamp      : Date

  - File upload validation (type and size limits)
  - Domain-restricted AI to prevent misuse and data leakage
